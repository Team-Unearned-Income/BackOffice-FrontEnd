# QA 수정 내역 (2026-07-18)

로컬 백엔드(`knockin-be-local`, H2 파일 DB)를 대상으로 BO 전 화면 API 연동을 Playwright로 직접 눌러보며 QA한 결과 발견/수정한 버그 정리. BE 수정 20건은 `11th-1team-BE` 레포, FE 수정 1건은 이 레포 기준.

## 1. CORS `allowedMethods`에 PATCH 누락 (치명적)

- **파일**: `SecurityConfig.java`
- **증상**: 회원 정지 해제, 신원인증 승인/반려, admin 권한 부여/회수 등 **PATCH를 쓰는 모든 처리 버튼이 403 "Invalid CORS request"로 실패**.
- **원인**: `corsConfigurationSource()`의 `allowedMethods`가 `GET, POST, PUT, DELETE, OPTIONS`만 포함하고 PATCH가 빠져 있었음. Vite dev 프록시가 브라우저의 `Origin` 헤더를 그대로 백엔드에 전달하는데, Spring의 `CorsFilter`는 Origin 헤더가 있으면 (프록시를 거쳐 실제로는 같은 브라우저 세션이라도) preflight 여부와 무관하게 매 요청마다 CORS 규칙을 검증함 — 이때 method가 allowedMethods에 없으면 즉시 403.
- **왜 지금까지 안 걸렸나**: BO 화면에서 PATCH를 쓰는 처리(승인/반려/정지해제 등)를 실제로 눌러본 적이 없어서 발견이 늦었음. GET/POST/PUT 흐름만으로는 재현 안 됨.
- **수정**: `allowedMethods`에 `"PATCH"` 추가.

## 2. 신원인증 "검토 대기" 리스트가 영구히 비어있음 (치명적)

- **파일**: `AuthenticationServiceImpl.java`, `AuthenticationApprove.java`
- **증상**: 이메일 인증을 완료해도 BO "신원 인증 관리 > 검토 대기" 탭에 절대 아무것도 안 뜸.
- **원인**: `AuthenticationApprove`(승인/반려 상태를 담는 테이블, `authentication_id`가 PK인 1:1 관계)에 **`PENDING` 상태 레코드를 생성하는 코드가 어디에도 없었음**. 대기 목록 쿼리는 `authenticationApprove.status = PENDING`인 행을 찾는데, 그 행 자체가 생성되는 지점이 없어서 항상 빈 리스트.
- **부가 문제**: 기존 BO 승인/반려 로직(`saveVerifications`/`deleteVerifications`)이 `AuthenticationApprove.builder()...save()`로 **새 레코드를 insert**하는 방식이었음. `authentication_id`가 PK인 1:1 관계라 PENDING 생성 로직을 추가하면 승인/반려 시 같은 PK로 중복 insert가 나서 제약 위반이 날 상황이었음.
- **수정**:
  - 이메일 인증 성공(`confirmAuthNumCompany`/`confirmAuthNumStudent`) 시점에 `AuthenticationApprove(status=PENDING)`을 생성하도록 추가.
  - `AuthenticationApprove`에 `approve()`/`reject()` 메서드를 추가하고, BO 승인/반려는 기존 PENDING 행을 `findByAuthentication`으로 찾아 **상태만 변경**하도록 수정(신규 insert 방식 → update 방식).

## 3. `findBackOfficeInquirie` (문의 상세) — id로 필터링을 안 함

- **파일**: `InquiryRepositoryImpl.java`
- **증상**: `/bo/inquiries/{id}`에 **어떤 id를 넣어도 조건에 맞는 임의의 문의 하나가 반환됨** (요청한 문의가 아닐 수 있음).
- **원인**: QueryDSL where절에 `inquiry.id.eq(id)` 조건이 빠지고 `inquiry.isDeleted.eq(false)`만 있었음.
- **수정**: `.where(inquiry.id.eq(id), inquiry.isDeleted.eq(false))`로 조건 추가.

## 4. `@ColumnDefault`만 있고 `@Builder.Default` 누락 → NULL 저장 (13개 엔티티)

- **공통 원인**: JPA/Hibernate에서 `@ColumnDefault`는 **DB가 컬럼을 아예 명시하지 않고 insert할 때만** 적용되는 DDL 힌트다. Hibernate는 매핑된 필드를 항상 insert 문에 포함시키므로, `Boolean` 필드에 `@Builder.Default`(또는 생성자에서 명시적 값 설정)가 없으면 `builder().build()` 시 `null`이 그대로 DB에 저장된다. `is_deleted` 같은 컬럼은 이후 조회 쿼리가 `WHERE is_deleted = false`로 필터링하는데, SQL 3치 논리상 `NULL = false`는 거짓이라 **그 행은 영원히 조회 결과에서 사라진다**. `NOT NULL` 제약이 걸린 컬럼(`InquiryComment`, `Faq`, `LifePattern`, `RoomType` 등)은 아예 insert 시점에 500 에러로 터진다.
- **실제 QA로 재현되어 고친 것**:
  - `Inquiry.isDeleted` — 문의 작성해도 목록에 안 뜸
  - `InquiryComment.isDeleted` — 문의 답변 전송 시 500 (NOT NULL 위반)
  - `Notification.isDeleted` — 공지 작성해도 목록에 안 뜸
  - `Faq.isDeleted` — FAQ 저장 경로에 영향
  - `LifePattern.isDeleted` — 생활패턴 항목 추가 시 500
  - `RoomType.isDeleted` — 방 유형 추가 시 500
  - `Authentication.isAccepted` / `isDeleted` — 신원인증 흐름 안정성
- **QA로는 못 만났지만 같은 패턴이라 예방적으로 같이 고친 것** (해당 기능이 미완성/화면 없음이라 실사용 경로로 재현은 안 됨, 하지만 언젠가 켜지면 100% 동일하게 터짐):
  - `Agreement.isDeleted`, `Alarm.isRead`, `AlarmSetting.isEnabled`, `ChatRoomMember.isLeft`, `InquiryCategory.isDeleted`, `MyRoommate.isDeleted`, `RoomExtraOption.isDeleted`, `RoommateCalendar.isDeleted`, `RoommateCalendarCategory.isDeleted`, `AppVersion.isDeleted`, `AuthEmail.isDeleted`, `MemberPrivacy.type`(기본값 `PUBLIC`)
- **손대지 않은 것**: `Block.isDeleted` — `Block` 엔티티가 코드베이스 어디서도 생성되지 않는 미구현 기능이라 (컨트롤러가 스텁 상태) 지금 고쳐도 검증 불가능하고 리스크만 있어 제외.
- **수정 방식**: 각 필드에 `@Builder.Default` 추가 + 필드 선언부에 기본값 명시(`= false`, `= true`, `= MemberPrivacyType.PUBLIC` 등). `@ColumnDefault`는 스키마 문서화 목적으로 유지.

## 5. 필드 누락/오탈자로 응답값이 항상 null

- **`BoVerificationWaitingListDto.EmployeeAuthItem`에 `id` 필드 자체가 없었음** → BO 화면에서 승인/반려 버튼을 눌러도 `PATCH /wait/undefined/approve`로 요청이 나가 무조건 실패. `id` 필드 추가로 해결.
- **문의/신원인증 목록·상세 4곳**(`findBackOfficeInquirieList`, `findBackOfficeInquirie`, `findVerificationsList`, `findVerifications`, `findVerificationApproves`, `findVerificationCancels`) — QueryDSL 프로젝션에서 `inquiry.createdAt` / `authenticationApprove.createdAt`을 그대로 매핑했는데, DTO 필드명이 `createAt`(오타, d 없음)이라 `Projections.fields`가 이름 불일치로 값을 못 채우고 항상 null. FE가 이미 `createAt` 기준으로 짜여 있어서(3개 화면 이상에서 사용 중) **FE를 고치는 대신 BE 쿼리에 `.as("createAt")` 별칭을 붙여 맞춤**.

## 6. `FaqServiceImpl.saveFaq` — 작성자(member)를 안 넘김

- **증상**: FAQ 저장 시 500 (`member_id` NOT NULL 위반).
- **원인**: `memberService.findById()`로 로그인한 관리자를 조회는 해놓고, 정작 `Faq.builder()`에 `.member(member)`를 안 넘김.
- **수정**: `Faq.builder().member(member)...` 추가.

## 7. `BoMemberDetailDto.createdAt` 타입 불일치 — 최초 발견한 500 (에러코드 3005)

- **증상**: 회원 상세 조회(`/bo/member/{id}`)가 500.
- **원인**: 엔티티(`CreatedAtEntity.createdAt`)는 `LocalDateTime`인데 DTO 필드는 `LocalDate`로 선언되어 있어, QueryDSL `Projections.fields`가 타입 불일치로 `IllegalArgumentException`을 던짐.
- **수정**: DTO 필드 타입을 `LocalDate` → `LocalDateTime`으로 변경.

## 8. FE — `/posts` 상세 패널 진입 시 크래시

- **파일**: `PostList.vue`, `PostDetailPanel.vue`
- **증상**: 게시글 상세를 열면 `Cannot read properties of undefined (reading 'bg')`로 전체 화면 크래시.
- **원인**: FE가 `post.isDeleted`를 읽어 상태 뱃지 색상(`POST_STATUS_META[...]`)을 찾는데, 실제 API 응답 필드명은 `deleted`(Jackson이 boolean getter `isDeleted()`를 직렬화할 때 `is` 접두사를 떼고 `deleted`로 내보내는 규칙 때문). 존재하지 않는 키로 조회해 `undefined.bg` 접근 → 크래시.
- **수정**: `PostList.vue`(목록 컬럼/필터), `PostDetailPanel.vue`(상태뱃지/액션 노출 조건) 모두 `deleted` 기준으로 통일.
